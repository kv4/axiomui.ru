Legacy Drupal в банковском Kubernetes

Банковский сектор · 2025–2026 · Docker + k8s + rootless

Суть: Legacy-проект на Drupal 10 развёрнут в банковском Kubernetes с соблюдением всех требований безопасности: rootless-контейнеры, многоэтапная сборка, ротация секретов, fail-fast сценарии. Деплой ускорился с 3 часов до 8 минут.

«Деплой из 3+ часов ручной работы по SSH превращён в CI/CD-пайплайн за 8 минут: docker build → push → kubectl apply — zero-downtime через rolling update.»

«Многоэтапная сборка (6 этапов) + L2-кеширование BuildKit сократили образ на 60%. Сборочные инструменты и артефакты CI не попали в production.»

Контекст и проблема

Исходная система

Платформа корпоративной автоматизации (RPA) на Drupal 10, PHP 8.2, PostgreSQL. До контейнеризации разворачивалась вручную на выделенном сервере через SSH.

Как выглядел деплой:

SSH → git pull → rm -rf vendor/ → composer install → drush sql-drop → импорт SQL → drush cr → drush updb → drush migrate → drush cron

Ключевые проблемы:

  • Деплой 2–3 часа — composer install без кэша тянул зависимости из репозиториев
  • Downtime при деплое — сайт недоступен на время сброса и импорта БД
  • Ручное управление конфигами — пароли, ключи, соли — всё в settings.php
  • Невоспроизводимость окружения — dev-сервер отличался от production
  • Одна точка отказа — одна машина, без масштабирования и изоляции

Техническое решение

Docker: многоэтапная сборка (6 этапов)

Сборка разделена на изолированные этапы: apt-подготовка (ключ nginx), composer install, компиляция PHP-расширений (imagick через PIE), npm build для React-дашборда, системный фундамент (nginx + supervisor), финальный образ.

Двухуровневое кэширование BuildKit: L1 — слои Docker, инвалидируются через CACHE_DATE; L2 — персистентные кэши для apt и composer. Образ сокращён на ~60% за счёт исключения сборочных инструментов.

Rootless: переезд на порт 8080

Все сервисы перенесены на порт 8080. Nginx настроен на absolute_redirect off; port_in_redirect off; — иначе rootless-порт 8080 просачивался в URL при редиректах.

12-factor config: самописная библиотека

Для Drupal, который штатно хранит всё в settings.php, написана библиотека environment-loader — 13 PHP-провайдеров, которые собирают конфигурацию из env-переменных и /run/secrets/. Поддерживается 40+ переменных: от подключения к БД до настроек Keycloak и Prometheus. Fail-fast в каждом провайдере — если нет HASH_SALT или пароля БД, приложение не стартует.

Fail-fast и Health Checks

Kubernetes probes на трёх уровнях: liveness (проверка ответа приложения), readiness (готовность принимать трафик), startup (ожидание инициализации). Docker HEALTHCHECK — дублирование на уровне контейнера. Entrypoint-скрипты с set -e — 9 последовательных этапов инициализации, каждый с fail-fast.

Деплой в банковском k8s

Приложение разделено на два сервиса: ideahub-gateway (nginx, NodePort 32599) и ideahub-backend (PHP-FPM, ClusterIP). Файлы хранятся в S3 (MinIO), конфигурация синка — внутри образа. Zero-downtime через rolling update с maintenance mode на время миграций БД.
